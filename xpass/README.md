# SkillCorner xPass Pipeline

This directory contains the first xPass workflow for SkillCorner K League
Dynamic Events.

The initial implementation follows the Anzer & Bauer expected-pass framing:
estimate pass completion probability from the pass location, intended target,
pass geometry, and surrounding event context. It intentionally starts with
SkillCorner Dynamic Events rather than a full ball-trajectory/player-reach
physics model. SkillCorner's supplied xPass is kept aside as a benchmark.

See the repo-level [README](../README.md) for data sources (open vs licensed) and the
tutorial-vs-analysis split.

## Published report + vendors

- **SkillCorner (published)** — uses SkillCorner's *provided* xPass directly. The published
  `week2-xpass` post figures are regenerated by `week2_report.py`
  (reads `tmp/data/skillcorner_xpass/passes.parquet` → `reports/skillcorner_week2/`).
- **Bepro (builder only)** — `bepro_drive_passes.py` builds a Bepro pass table, kept so future
  analysis can use Bepro data. A *publishable* Bepro xPass model is future work: Bepro's
  `relative_event` is the **realized** next touch (not the intended receiver), so
  destination-derived features leak the outcome — a proper model needs an intended-target
  estimate. (The earlier event-only Bepro model/report were removed as unpublishable duplicates.)

## Tutorial (open data)

`notebooks/00_xpass_from_scratch.ipynb` builds xPass from scratch on public SkillCorner Open
Data (`DATA_SOURCE` toggle, default `"opendata"`): pass table → distance/progression/angle
features → XGBoost → calibration → **comparison against SkillCorner's provided xPass** → PAx,
then verifies it matches the modules above. The existing
`notebooks/week2_xpass_analysis.ipynb` is a pipeline run on licensed data, not a
from-scratch tutorial.

## Directory Layout

```text
xpass/
  skillcorner_passes.py              # build the SkillCorner pass table
  xpass_features.py                  # pass-level feature engineering
  train_skillcorner_xpass.py         # model training and SkillCorner comparison
  week2_report.py        # regenerate the published week2-xpass figures (SkillCorner)
  bepro_drive_passes.py              # Bepro pass builder (vendor data capability; publishable model = future work)
  notebooks/                         # two notebooks:
    00_xpass_from_scratch.ipynb      # (1) tutorial + simple analysis on open data (DATA_SOURCE toggle)
    week2_xpass_analysis.ipynb       # (2) reproduce the published week2-xpass analyses (SkillCorner pipeline)
```

Generated data/models (`tmp/data/skillcorner_xpass/`) and report outputs (`reports/`) are
gitignored; the published week2-xpass figures live on the KAISport site.

## 1. Build Pass Table

```bash
python -m xpass.skillcorner_passes \
  --skillcorner-root /data2/MHL/data/skillcorner/kleague \
  --season-names 2023 2024 2025 \
  --out tmp/data/skillcorner_xpass/passes.parquet
```

Passes are defined as:

```text
event_type == "player_possession" and end_type == "pass"
```

The binary target is:

```text
pass_completed = pass_outcome == "successful"
```

Offside pass outcomes are kept in the raw table but excluded from model
training by default.

The initial model also requires passer and targeted-receiver coordinates. Rows
without an identified target are better handled later by a separate blocked-pass
or target-estimation extension.

## 2. Add Features

Feature engineering is implemented in `xpass_features.py`. The training script
calls it automatically, but it can also be run directly:

```bash
python -m xpass.xpass_features \
  --passes tmp/data/skillcorner_xpass/passes.parquet \
  --out tmp/data/skillcorner_xpass/passes_with_features.parquet
```

The custom model uses a strict pre-pass feature set. It excludes SkillCorner's
own `skillcorner_xpass`, `player_targeted_xthreat`, `dangerous`, and
`difficult_pass_target` fields. It also excludes obvious
post-outcome/matching-quality fields such as pass reception matching, `*_end`
features, defensive-line gain features, and actual line-break outcome fields.
Those fields remain available for diagnostics and comparison.

## 3. Train And Compare xPass Models

```bash
python -m xpass.train_skillcorner_xpass \
  --passes tmp/data/skillcorner_xpass/passes.parquet \
  --train-seasons 2023 2024 \
  --test-seasons 2025 \
  --models logistic xgboost
```

Outputs:

```text
tmp/data/skillcorner_xpass/skillcorner_xpass_best.joblib
tmp/data/skillcorner_xpass/skillcorner_xpass_logistic.joblib
tmp/data/skillcorner_xpass/skillcorner_xpass_xgboost.joblib
tmp/data/skillcorner_xpass/scored_passes.parquet
tmp/data/skillcorner_xpass/scored_passes_logistic.parquet
tmp/data/skillcorner_xpass/scored_passes_xgboost.parquet
tmp/data/skillcorner_xpass/metrics.json
tmp/data/skillcorner_xpass/model_comparison.csv
tmp/data/skillcorner_xpass/skillcorner_comparison.csv
tmp/data/skillcorner_xpass/team_pax.csv
tmp/data/skillcorner_xpass/player_pax.csv
```

`team_pax.csv` and `player_pax.csv` follow the Opta Analyst PAx framing:

```text
PAx = completed passes - expected completed passes
PAx per 100 = PAx / passes * 100
```

## Tutorial Notebook

Open `notebooks/week2_xpass_analysis.ipynb` for a tutorial-style
walkthrough:

1. Build or load the SkillCorner pass table.
2. Inspect pass outcomes and SkillCorner xPass coverage.
3. Add custom expected-pass features.
4. Train a custom xPass model.
5. Compare custom xPass with SkillCorner's supplied xPass.
6. Summarize team/player passing performance.
